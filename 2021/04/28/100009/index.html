vue项目性能优化

前言

我们使用vue框架能快速、方便的开发单页面应用，但开发的项目中也会存在一些问题，如首屏白屏时间过长，加载包过多打包太久等等问题。本文针对这些问题，通过总结以往项目经验及参考部分文档做的一些总结，总结内容由三部分组成：

  • Vue框架代码层面的优化
  • webpack(或vue-cli)打包配置层面的优化
  • web技术层面的优化

一、Vue框架代码层面的优化

1.2 图片懒加载

针对图片资源过多的页面，为了加快页面加载速度，图片使用懒加载方式是有必要的。针对图片懒加载具体原理，简要说明就是图片未在可视区时，暂时不加载，将图片资源放置在data-url自定义的属性中，src设置未空，待页面滑动到图片可视区位置，在去data-url值赋值给src，使图片正常显示处理即可。

1.3 v-if与v-show区分使用场景

v-if 是“真正”的条件渲染，因为它会确保在切换过程中条件块内的事件监听器和子组件适当地被销毁和重建。值为false相当于display: none。

v-if 也是惰性的：如果在初始渲染时条件为假，则什么也不做——直到条件第一次变为真时，才会开始渲染条件块。

一般来说，v-if 有更高的切换开销，而 v-show 有更高的初始渲染开销。因此，如果需要非常频繁地切换，则使用 v-show 较好；如果在运行时条件很少改变，则使用 v-if 较好。

针对页面子组件太多的情况，切换比较少的组件推荐使用v-if，减少页面进入时需要渲染太多子组件而时白屏时间过长。

1.4 v-for必须配置key，且避免与v-if同时使用

  1. v-for配置key
    建议尽可能在使用v-for时提供key，除非遍历输出的 DOM 内容非常简单，或者是刻意依赖默认行为以获取性能上的提升。

如果不使用 key，Vue 会使用一种最大限度减少动态元素并且尽可能的尝试就地修改/复用相同类型元素的算法。而使用 key 时，它会基于 key 的变化重新排列元素顺序，并且会移除 key 不存在的元素。

有相同父元素的子元素必须有独特的 key。重复的 key 会造成渲染错误

  1. v-for避免与v-if同时使用
    同一节点v-for 的优先级比 v-if 更高,如果两者同时使用，这将会影响性能，因为如果列表即使只有一小部分满足条件，vue也得每次重渲染的时候遍历整个列表。
    推荐将原数组或对象更换为计算属性。
    这样的好处：
  • 过滤后的列表只会在 users 数组发生相关变化时才被重新运算，过滤更高效。
  • 使用 v-for=”user in activeUsers” 之后，我们在渲染的时候只遍历活跃用户，渲染更高效。
  • 解耦渲染层的逻辑，可维护性 (对逻辑的更改和扩展) 更强。

1.5 watch与computed区分使用场景

computed：

  1. 支持缓存，只有依赖数据发生改变，才会重新进行计算
  2. 不支持异步，当computed内有异步操作时无效，无法监听数据的变化
  3. computed 属性值会默认走缓存，计算属性是基于它们的响应式依赖进行缓存的，也就是基于data中声明过或者父组件传递的props中的数据通过计算得到的值
  4. 如果一个属性是由其他属性计算而来的，这个属性依赖其他属性，是一个多对一或者一对一，一般用computed
  5. 如果computed属性属性值是函数，那么默认会走get方法；函数的返回值就是属性的属性值；在computed中的，属性都有一个get和一个set方法，当数据变化时，调用set方法。

demo:

watch:

  1. 不支持缓存，数据变，直接会触发相应的操作；
  2. watch支持异步；
  3. 监听的函数接收两个参数，第一个参数是最新的值；第二个参数是输入之前的值；
  4. 当一个属性发生变化时，需要执行对应的操作；一对多；
  5. 监听数据必须是data中声明过或者父组件传递过来的props中的数据，当数据变化时，触发其他操作，函数有两个参数，
  • immediate：组件加载立即触发回调函数执行，
  • deep: 深度监听，为了发现对象内部值的变化，复杂类型的数据时使用，例如数组中的对象内容的改变，注意监听数组的变动不需要这么做。注意：deep无法监听到数组的变动和对象的新增，参考vue数组变异,只有以响应式的方式触发才会被监听到

1.6 监听事件的销毁

Vue组件销毁时，会自动清理它与其他实列的连接，解绑它的全部指令及自身的事件监听器，而不能销毁js内绑定的监听事件，所以推荐将绑定的事件在组件销毁前销毁

1.7 合理使用keep-alive组件

官网解释： 包裹动态组件时，会缓存不活动的组件实例，而不是销毁它们。和 相似， 是一个抽象组件：它自身不会渲染一个 DOM 元素，也不会出现在父组件链中。 当组件在 内被切换，它的 activated 和 deactivated 这两个生命周期钩子函数将会被对应执行。 在 2.2.0 及其更高版本中，activated 和 deactivated 将会在 树内的所有嵌套组件中触发。 主要用于保留组件状态或避免重新渲染。

keep-alive的生命周期

  • 初次进入时：created > mounted > activated；退出后触发 deactivated
  • 再次进入：会触发 activated；事件挂载的方法等，只执行一次的放在 mounted 中；组件每次进去执行的方法放在 activated 中。

参考知乎链接：https://zhuanlan.zhihu.com/p/96740001

1.8 长列表性能优化

  • 针对数据不更新的大数据对象或数组，可以使用Object.freeze(obj)方法将对象冻结起来，减少不必要的引用消耗
  • 如果产品需求需要展示非常长或无限长的列表，可参照vue-virtual-croll-list和vue-virtual-scroller有优化该类场景（原理基本时只渲染可视区范围的dom节点，减少内存的消耗）

三、web技术层面的优化

3.1 开启gzip压缩

gzip开启gzip压缩可非常有效加快页面加载，根据以往经验，开启压缩与未开启压缩文件大小相差几倍，速度相应也能提升几倍，生产环境开启gzip基本是必须的
nginx开启gzip配置如下：

检查是否压缩开启成功，再配置完成后重启服务，可通过浏览器调试页面观察network面版response header中accept-encoding: gzip

3.2 使用缓存

为了提高用户加载页面速度，对静态资源使用缓存是非常有必要的，针对不同类型资源，配置不同状态的缓存（强缓存、协商缓存），能非常有效提升页面加载速度，缓存可通过响应头header来配置，涉及的知识点比较多，可以深入研究点比较多，这里暂时不做详细解读。

3.3 静态资源使用CDN

CDN有更好的可用性，更低的网络延迟和丢包率，图片等静态资源使用CDN，可以非常有效的提升网站的加载速度
